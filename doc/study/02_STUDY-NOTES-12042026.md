## What I Studied Today

### 1. Project Shape

- This is a multi-module Maven project centered on IAM protocol learning, not just app delivery.
- The backend is split by protocol/domain:
  - `auth-core`
  - `oauth-oidc`
  - `saml-federation`
  - `scim`
  - `mfa`
  - `device-flow`
  - `demo-resource`
  - `api-gateway`
- The main Spring Boot entry point is:
  - `backend/api-gateway/src/main/java/com/iam/gateway/ApiGatewayApplication.java`

### 2. Core Architecture Insight

- `auth-core` is the shared persistence layer.
- It contains the JPA entities and repositories used by the other modules.
- The system is easiest to understand as:
  - protocol concepts
  - mapped into tables
  - exposed through controllers
  - implemented through services

### 3. Key Schema Mental Model

The `oauth_client` table stores client registration metadata:

- `client_id`
- `client_secret_hash`
- `redirect_uris`
- `allowed_scopes`
- `grant_types`
- `is_public`

Source:
- `backend/auth-core/src/main/resources/db/migration/V1__init.sql`
- `backend/auth-core/src/main/java/com/iam/authcore/entity/OAuthClient.java`

Important detail:

- `redirect_uris`, `allowed_scopes`, and `grant_types` are stored as `TEXT`
- They are intended to be comma-separated values, not JSON columns

## What I Learned About `allowed_scopes`

- `allowed_scopes` is client-specific metadata stored in `oauth_client.allowed_scopes`
- It is enforced during authorization request validation
- In `AuthorizeService`, requested scopes are checked against the client's allowed scopes

Relevant file:
- `backend/oauth-oidc/src/main/java/com/iam/oauth/service/AuthorizeService.java`

Current server-supported scopes exposed in discovery:

- `openid`
- `profile`
- `email`

Relevant file:
- `backend/oauth-oidc/src/main/java/com/iam/oauth/controller/DiscoveryController.java`

## What I Learned About `grant_types`

- `grant_types` is also client-specific metadata stored in `oauth_client.grant_types`
- The discovery document says the server supports:
  - `authorization_code`
  - `client_credentials`
  - `refresh_token`
  - `urn:ietf:params:oauth:grant-type:device_code`

Important observation:

- I did not find enforcement of `client.getGrantTypes()` in the token flow
- So `grant_types` currently looks more like stored metadata than a fully enforced authorization rule

Relevant files:
- `backend/oauth-oidc/src/main/java/com/iam/oauth/controller/DiscoveryController.java`
- `backend/oauth-oidc/src/main/java/com/iam/oauth/service/TokenService.java`

## Quick Q&A

### Purpose of `oauth_client`

- Represents a registered application that can use the authorization server
- Stores:
  - identity: `client_id`
  - authentication mode: `client_secret_hash`, `is_public`
  - redirect safety: `redirect_uris`
  - permissions: `allowed_scopes`
  - intended flows: `grant_types`
- Used to validate who is calling, where redirects may go, and what the app may request

Enforced in:
- `backend/oauth-oidc/src/main/java/com/iam/oauth/service/AuthorizeService.java`
- `backend/oauth-oidc/src/main/java/com/iam/oauth/service/TokenService.java`

### Is `redirect_uris` the same across all clients?

- No
- `redirect_uris` belongs to each individual OAuth client
- Each client has its own allowed callback URL list
- During authorization, the requested `redirect_uri` must exactly match one of the URIs registered for that specific client
- This prevents redirecting authorization codes to another app's callback URL

### Real-life onboarding and redirect URIs

- Yes, a new client/app usually needs its own registered redirect URI
- The redirect URI points back to that app after login
- One client may have multiple redirect URIs for different environments:
  - local
  - staging
  - production
- These URIs should be explicitly registered and exact-matched by the authorization server

### Do we pass callback URLs in the request?

- Usually yes, the client sends `redirect_uri` in the authorization request
- The server does not trust it blindly
- It checks whether that `redirect_uri` exactly matches one of the URIs registered for that client
- So:
  - passed by the client at runtime
  - validated against server-side client registration

### Why pass `redirect_uri` if it is already registered?

- Because one client may have more than one valid callback URL
- The request tells the authorization server which exact callback URL to use for this login transaction
- Main design concerns:
  - correctness: send the user back to the intended app endpoint
  - security: only allow pre-registered exact-match URLs
  - binding: use the same `redirect_uri` across authorize and token exchange
- Without validation, an attacker could try to inject a malicious callback URL and steal the authorization code

### Why PKCE exists

- PKCE protects the authorization code flow, especially for public clients
- Problem:
  - an attacker may intercept or steal the authorization code
  - public clients cannot safely keep a client secret
- Solution:
  - client sends `code_challenge` during authorization
  - later sends `code_verifier` during token exchange
  - server checks they match
- Result:
  - stolen auth code alone is not enough to get tokens

### What `state` is in the request

- `state` is a client-generated value sent in the authorization request
- the authorization server returns it unchanged in the redirect response
- main purpose:
  - prevent CSRF in the OAuth redirect flow
- secondary purpose:
  - let the client restore app context after login
  - example: remember which page or action the user started from
- important:
  - `state` is not for token security
  - PKCE protects the authorization code
  - `state` protects the client redirect flow
